1 · Problem / Motivation

Heute werden Sessions als Grid-Mosaik (CSS-grid, Spaltenzahl dynamisch nach Fensterbreite) oder im Single-View (eine voll) gezeigt. Bei kleiner Auflösung → 1 Spalte → lange vertikale Liste. Gewünscht: VS-Code-artige Alternativen — Tabs und (später) abkoppelbare Fenster — ohne das Legacy-Verhalten zu verlieren (per Setting wählbar).

Phase 2 — Settings-Fenster Pop-out ✅ erledigt

Das Einstellungsfenster als eigenständiges OS-Fenster abkoppelbar machen. Bewusst vor Detach: dasselbe Mehrfenster-Muster, aber ohne den heiklen terminal-data-Byte-Stream → validiert die Mechanik (2. BrowserWindow, IPC, Live-Apply-Broadcast) als risikoarmer Proof-of-Concept.

2.1 IST & Prinzip

  • Heute: Settings = DOM-Overlay im Main-Renderer (settings-panel.js, #settings-viewer) — kein eigenes Fenster.

Phase 3 — Detach / abkoppelbare Session-Fenster großer Umbau

Eine Session in ein eigenes OS-Fenster auslagern (wie VS Code „Move into New Window") — auf einen zweiten Monitor ziehbar, Session läuft weiter. Eigener Branch, erst nach Phase 1 + 2 (nutzt das in Phase 2 erprobte Mehrfenster-Muster).

3.1 Grundprinzip

Eine Session wird entweder im Main-Fenster oder in genau einem Detach-Fenster gerendert — nie beides (sonst doppeltes Rendern + doppelter Input-Echo). Der PTY bleibt unverändert im Main-Prozess (activeSessions); nur welches Fenster den Byte-Stream bekommt, ändert sich.

3.2 Fenster-Registry im Main

// main.js
const detachedWindows = new Map(); // sessionId → BrowserWindow

function windowForSession(sessionId){
  const w = detachedWindows.get(sessionId);
  return (w && !w.isDestroyed()) ? w : mainWindow;
}

3.3 terminal-data routen (der zentrale Eingriff)

Den hardcoded Send (main.js:2074) ersetzen:

Vorher

mainWindow.webContents.send(
  'terminal-data', sessionId, chunk);

Nachher

const win = windowForSession(sessionId);
if (win && !win.isDestroyed())
  win.webContents.send(
    'terminal-data', sessionId, chunk);
Wichtig — nur den Byte-Stream routen. Sidebar-/Status-Events (Attention, busy-state, exit, OSC-Badges) müssen weiterhin den Main erreichen (die Sidebar lebt dort). Also: terminal-data → besitzendes Fenster; Lifecycle/Attention → Main (ggf. zusätzlich ans Detach-Fenster). Diese Trennung sauber durchziehen, sonst verschwinden im Detach-Fall Attention-Badges aus der Sidebar.

3.4 IPC: detach / reattach

HandlerAblauf
detach-session
  1. Neues BrowserWindow erzeugen (Detach-Renderer, siehe 2.5).
  2. detachedWindows.set(sessionId, win).
  3. Main-Renderer anweisen, sein xterm für die Session freizugeben (destroySession, PTY bleibt) → Main rendert sie nicht mehr.
  4. Detach-Fenster lädt Renderer → ruft openTerminal(sessionId) → Reattach → Buffer-Replay → rendert.
reattach-session
(auch bei Detach-Fenster-Close)
  1. detachedWindows.delete(sessionId) → terminal-data fällt auf Main zurück.
  2. Detach-Fenster schließen.
  3. Main-Renderer benachrichtigen → Session wieder öffnen/zeigen (openSession/showSession, Reattach replayt Buffer).

3.5 Detach-Renderer — index.html wiederverwenden (empfohlen)

Empfohlen index.html im Detach-Modus

Fenster lädt index.html?detached=<sessionId>. Renderer erkennt den Flag → blendet Sidebar/Tabs/Grid/Header aus, rendert nur die eine Session voll.

  • + erbt automatisch alle Terminal-Fixes (windowsPty, Paste, Scroll, Maus-Reporting, Rechtsklick).
  • + wenig neuer Code.
  • − lädt die ganze App pro Fenster (Speicher) — für ein Power-Feature ok.

Alternative: minimaler detached.html

Eigene Seite, instanziiert ein xterm via createTerminalEntry.

  • + leichter (Speicher).
  • − dupliziert Wiring; läuft Gefahr, Globals/Fixes zu verpassen → Drift.

3.6 Input / Fokus / Clipboard

  • terminal-input ist sessionId-gekeyed → aus dem Detach-Fenster routet es schon korrekt an den PTY. Funktioniert ohne Änderung.
  • Clipboard (OSC-52), Rechtsklick-Menü, Selektion etc. laufen über dieselbe terminal-manager-Logik → erben.

3.7 Lifecycle & Edge-Cases

FallVerhalten
Detach-Fenster vom User geschlossen→ reattach-session: Session kehrt zum Main zurück (nicht verloren). PTY lief durchgehend.
PTY exit während detachedExit-Event: Detach-Fenster zeigt Exit; Main-Sidebar-Status aktualisiert (Status-Events bleiben am Main).
Main-Fenster geschlossen, Detach offenDetach-Fenster mitschließen (abhängig vom Main). Sonst Orphan-Fenster. Auf Win/Linux: app.quit() bei window-all-closed (main.js:2307) — Detach-Fenster zählen mit.

Doppel-DetachVerhindern: Detach-Button deaktiviert/grau, wenn schon in detachedWindows.
BoundsOptional: Position/Größe je Detach-Fenster persistieren.

3.8 Zusammenspiel mit Tabs (Phase 1)

Detachte Session bekommt im Main-Fenster einen „Ghost"-Tab (ausgegraut + Icon „⤢"): Klick fokussiert das Detach-Fenster (statt showSession); Close/Reattach am Tab verfügbar. So bleibt die Session in der Main-UI auffindbar.
